Workspaces

How Skyforge provisions repos + templates + workspaces.

Docs home Skyforge home Dashboard

What a Workspace is

  • A Skyforge Workspace provisions a Git repo and native run configuration.
  • Artifacts are stored in Skyforge object storage under a shared bucket with per-workspace prefixes.
  • Workspace clones are mirrored into OpenVSCode Server so users can edit and push changes.

Create a workspace

  1. Log in on Skyforge home (SSO).
  2. Skyforge automatically provisions a default workspace on first login.
  3. Use the Dashboard to select your workspace and create deployments.

Default templates

  • Tofu: Init, Plan, Apply (Apply requires owner/admin + confirmation).
  • Ansible: Run playbook.yml (a placeholder file is created in the repo).

Artifacts

  • Use the workspace row buttons to Upload/Download/Browse artifacts.
  • Uploads go through Skyforge Server (simple and works behind a single DNS name).

RBAC model

  • Owners: can edit membership and run templates (and are synced to Git as repo admins).
  • Editors: can run templates and are synced to Git with write access.
  • Viewers: read-only in Skyforge and are synced to Git with read access.
  • Optionally, map directory groups into Owner groups/Editor groups/Viewer groups for automatic access.

OpenVSCode Server workspaces

  • Shared clones (all workspaces): /var/lib/skyforge/workspaces/<slug> (legacy path)
  • Your clones (workspaces you can access): /var/lib/skyforge/users/<username>/<slug>

Pushing changes back to Git

  • Log in to Git (/git/) with SSO and create a Personal Access Token.
  • In OpenVSCode Server, configure your Git identity and authenticate with your token.
git config --global user.name "Your Name"
git config --global user.email "you@company.com"

# Option 1 (recommended): use a credential helper
git config --global credential.helper store
git pull
git push

# Option 2: set an authenticated remote (avoid sharing terminals/screens)
git remote -v
